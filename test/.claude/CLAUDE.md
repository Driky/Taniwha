# test/ — Test suite

Tests mirror the `lib/` directory structure. Every module in `lib/taniwha/` has a corresponding test in `test/taniwha/`, and every module in `lib/taniwha_web/` has a corresponding test in `test/taniwha_web/`.

## TDD workflow

1. Write a small batch of related tests (2-5 tests for one function or behaviour).
2. Run them — they should be red.
3. Write the minimum code to make them green.
4. Refactor if needed.
5. Repeat for the next batch.

Never write all tests for a module at once. Build incrementally.

## Test support modules

| File | Purpose |
|---|---|
| `test/support/fixtures.ex` | Sample XML-RPC response binaries for transport/RPC tests |
| `test/support/mock_connection.ex` | Mox mock definition for `Taniwha.SCGI.Connection` behaviour |
| `test/support/channel_case.ex` | Shared setup for channel tests |
| `test/support/conn_case.ex` | Shared setup for controller tests (generated by Phoenix) |

## Mocking strategy

We use **Mox** for all external boundaries:

- `Taniwha.SCGI.ConnectionMock` — mocks the socket connection behaviour. Used in RPC.Client tests.
- When testing Commands, mock the RPC.Client (via a behaviour or Mox).
- When testing Poller, mock Commands.
- When testing Channels/Controllers/LiveView, mock Commands.

**Rule:** Only mock at behaviour boundaries. Never mock internal pure functions — test those directly.

## Running tests

```bash
# Full suite
mix test

# Specific file
mix test test/taniwha/scgi/protocol_test.exs

# With coverage
mix test --cover

# Only tests tagged as integration (when rtorrent is available)
mix test --only integration
```

## CI checks (all must pass)

```bash
mix format --check-formatted
mix credo --strict
mix dialyzer
mix test
```
